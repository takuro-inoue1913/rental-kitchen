# 開発フロー

## ブランチ運用
- **main ブランチに直接 push しない**。必ず作業ブランチを切って PR 経由で merge する。

## Phase ごとの PR ルール
1. 各 Phase の実装が完了したらブランチを push し、PR を作成する
2. PR 作成後、GitHub Copilot にレビューをリクエストする
3. **PR が merge されるまで次の Phase に進まない**
4. レビュー指摘があれば **指摘1件につき1コミット** で修正する
5. 各指摘への返信は以下の構成で書く:
   - **理解**: レビュー指摘をどう理解したか
   - **解釈**: その問題がなぜ起きるか / どういうリスクがあるか
   - **対応**: 具体的にどう修正したか + commit URL（対応不要の場合はその理由）
6. 全指摘への対応完了後、再度レビューを受ける

## PR 作成後の検証
- PR 本文の **Test plan** に記載した項目をすべて実際に検証する（Vercel プレビューでの動作確認・コマンド実行など）
- 検証結果に問題があれば修正コミットを追加してから Copilot レビューに回す

## バグ修正はテスト駆動（TDD）で行う
1. まずバグを再現するテストを書く
2. テストが **失敗する** ことを確認する
3. 修正を実装する
4. テストが **通る** ことを確認する

## コミットの粒度
- 変更のコンテキスト（目的）ごとにコミットを分ける
- 異なる目的の変更を1つのコミットに混ぜない（例: バグ修正とドキュメント更新は別コミット）

## コンフリクト解消
- PR にコンフリクトが発生した場合、`git merge origin/main` で解消する（force push 禁止のため rebase は使わない）
- コンフリクト解消は独立したマージコミットとし、修正内容と混ぜない

## ドキュメント
- `docs/PHASES.md`: 全フェーズの実装計画・内容・ブランチ名。セッションが変わっても参照すること。
